Version Control System:GIT

Why Version control?
You're going to be working on a long project so you might difinitely make mistakes. Or you might also want to experiment. How do we do that safely?

  • By using GIT.

GIT CONFIG

The first thing to do is to configure Git with a couple of pieces of information it's going to need later.

%git config --global user.email sodaba.bashiri@powercoders.org
%git config --global user.name "Sodaba Bashiri"

So run these two commands, using your e-mail address and name.

Git fundamentals

Git commands copy files between these locations:

  1. Working directory
  2. Index
  3. Repository

Typical Git workflow

Git init - Git needs to know where the top, or root, of the repository is going to be. Once you do it with git init, Git can start to manage all the files in that directory, and all the directories below. You can have multiple Git repositories on your computer, as long as they are not inside one another.

%git init

Git add - When you're done with the file you're working on, you copy it to the index using git add.

%git add dir-name
...
%git commit -m "Write a message."
...
%git push

Git commit - When you've double checked your file in the index and you're sure to save it to the repository, you use git commit.

%git commit -m "Write a message."
...
%git push

To make a copy of the file, or files, in the index, and save them in the repository we use git commit. Git needs us to give a message that is associated with the commit. Here, I'm using the -m flag to specify the commit message. Don't forget the double quotes.

Git status - it shows the tracked or untracked, commited or uncommited files.

%git status

( SIDE NOTE) - The index is sometimes also known as the cache, or stage. So as well as reading documentation that talks about adding a changed file to Git, you might also see term staged being used.

Git rm - If you wanted to remove/delete a file from the index, you use git rm. ( This wouldn't delete it from the working directory, just the index. So before that, you have to delete the file from the computer.)

%git rm --cached<files>..."
...
%git commit -m "Write a message."

Git log - Shows you what's changed.

%git log

Git diff - Once we've made some changes that we want to add to the index, it can be helpful to easily view them to double check that they're what we want. For that we use the git diff command.

This shows the difference between what's in the working directory and what's in the index. If it didn't find any file in the index, it compares with the file in the repository.

%git diff

Unified diff format

          1: @@ -10,6 +10,5 @@
          2: <p>This was one of my example paragraphs that
          3:    runs over more than one line.</p>
          4:
          5: -<p>This paragraph will be deleted.</p>
          6: -
          7: -<p>This paragraph will be updated.</p>
          8: +<p>This paragraph will be updated
          9: +  with some extra content.</p>
          10:
          11:+<p>This paragraph is completely new.</p>
          12:+
          13: <p>This is another example paragraph.</p>
          14:
  • @@ is the hunk header
    • -m,n = Starts at line m, continues for n lines
    • +o,p = Starts at line o, continues for p lines
  • First column is what happened:
    • First column is what happened
    • Space This line is not changed
    • - = This line was deleted
    • + = This line was added

Git diff --cached - It compares the index with the repository.

%git diff --cached

Git mv - We use this to rename the file.

%git mv old-name new-name
...
%git commit -m "Write a message."

Distributed version control (DVCS)

So far your version control repository is only stored on your computer. Distributed version control was invented for teams of programmers, to make it easy for other people on the team to see your changes, maybe comment on them, or review them, or learn from them.

Essentially, this is a way of saying that two or more different repositories are linked together. Once they're linked together there are commands to manage copying commits from one repository to another. And lots of tools exist to let you see the repository in a web browser, view the commits, give feedback, and so on.

Various companies exist that provide online hosting for copies of your repositories, including GitHub, GitLab, BitBucket, and more. But we're going to use GitHub.

  • Create a GitHub account
    • https://www.github.com/join
    • Username
    • E-mail address
    • Password
    • Choose the free, public plan
  • Create a repository at GitHub
    • https://help.github.com/articles/create-a-repo/
    • Create a public repository
      • Called poco-yourname
  • Description: "Powercoders Basel: YOUR NAME"
  • Do not initialize with a README or LICENSE

Configure your local repository to sync

  • Other copies of your repository are called remotes
  • Find your GitHub repository URL
    • Looks like https://github.com/user/poco-yourname.git
  • git remote add origin https://github.com/user/poco-yourname.git
  • git remote -v to view configured remotes

In Git terminology, copies of your repository that are stored elsewhere, like on GitHub, are called remotes.

To tell your local repository about these we use the git remote add command. It needs to know the URL to the remote repository. For GitHub it's going to look like this URL here. Replace the user in the URL with the username that you chose when you signed up to Git, and the last part is poco.git because we called the repository poco.

This is also the URL for the repository in the browser, with .git added to the end, so you can cut and paste.

The origin in the command is the short-name that we're going to use for the remote. You can use anything you want. However, the convention in Git is that the public version of the repository is called origin

Push commits to GitHub

  • git push -u origin master - first time
  • git push - after that

Now that that your local repository is configured to sync with the repository on GitHub we can copy all of our commits so far to the copy of the repository on GitHub.

To do that, run the git push command.

The very first time we do this we have to tell Git exactly how these two repositories are linked, including how they branch. You will probably be prompted for the password for your GitHub account.
